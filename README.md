# Feature Envy Detection using Graph Neural Networks

A reproducible pipeline for detecting and refactoring Feature Envy code smells
using Graph Neural Networks, based on the paper:

> "Efficient Feature Envy Detection and Refactoring Based on Graph Neural Network"
> Yu et al., Automated Software Engineering (2025)
> DOI: https://doi.org/10.1007/s10515-024-00476-3

**Course:** DSCI 644.02 — Software Engineering for Data Science  
**Team:** Group 4

---

## What is Feature Envy?

Feature Envy is a code smell where a method interacts more with an external
class than its own class — suggesting the method belongs elsewhere. This
pipeline automates:

1. **Detection** — identifying which methods have feature envy
2. **Refactoring recommendation** — suggesting which class the method should
   move to (Move Method refactoring)

## Running the Pipeline

The pipeline consists of four sequential steps. Each step must be
completed before the next can begin.

---

### Step 1 — Data Exploration (optional)

Before preprocessing, explore the raw data to understand column names,
check for nulls, and verify ID alignment across files.

```bash
jupyter notebook notebooks/01_data_exploration.ipynb
```

Key checks performed:
- Column names in each CSV file
- Null values across all files
- Label distribution (positive rate per project)
- ID consistency across `ground_truth.csv`, `metrics.csv`,
  and `method-invocate-method.csv`

---

### Step 2 — Preprocess

Converts raw CSVs into PyTorch Geometric graph objects.
Must be run once before training. Reads from `data/raw/`
and writes to `data/processed/`.

```bash
# Process a single project
python scripts/preprocess.py --project activemq

# Process all projects
python scripts/preprocess.py --all
```

What this does:
1. **Method index** — maps every method ID to a consistent integer
   index 0 to N-1, saved as `method_to_idx.pt`
2. **Feature matrix X** — extracts 7 code metrics per method,
   normalizes with StandardScaler, saved as `X.pt`
3. **Graph construction** — builds edge index from method call
   relationships, zero edges dropped across all projects
4. **Label vector** — binary labels from `ground_truth.csv`,
   source and target class IDs stored separately
5. **Train/val/test splits** — stratified 60/20/20 splits across
   3 random seeds, saved as separate `graph.pt` files per seed

Expected output per project:
```
Total methods indexed: 15482
Smelly methods: 585 / 15482 (3.78%)
Methods matched to index: 15482
Methods in metrics but not in index: 0
Valid edges: 34054
Dropped edges: 0
Smelly methods where source != target: 585 / 585 (expect all)
Clean methods where source == target: 14897 / 14897 (expect all)
```

---

### Step 3 — Heuristic Baseline

Runs a deterministic rule-based detector to establish a performance
floor before any learned model is introduced. Does not require
preprocessing output — reads directly from `data/raw/` via the
saved graphs in `data/processed/`.

```bash
# Run on a single project
python scripts/baseline.py --project activemq

# Run on all projects with optimal threshold
python scripts/baseline.py --all --threshold 0.6
```

Detection rule:
```
score = NCMEC / (NCMEC + NCMIC + epsilon)
flag as smelly if score > threshold
```

A threshold sweep on the validation set identified 0.6 as optimal
by F1-score across most projects.

Baseline results (mean across 3 seeds, threshold=0.6):

| Project | Precision1 | Recall1 | F1-Score1 | Refactoring |
|---------|-----------|---------|-----------|-------------|
| ActiveMQ | 11.22% | 53.85% | 18.57% | 0.00% |
| Alluxio | 28.70% | 64.11% | 39.64% | 0.00% |
| BinNavi | 13.91% | 85.93% | 23.94% | 0.00% |
| Kafka | 14.60% | 59.10% | 23.41% | 0.00% |
| Realm-java | 30.61% | 29.23% | 29.85% | 0.00% |
| **Average** | **19.81%** | **58.44%** | **27.08%** | **0.00%** |

Refactoring is 0% by design — the heuristic has no access to call
graph structure and cannot predict the specific target class.

---

### Step 4 — Train GNN Model

Trains the SCG (SMOTE Call Graph) model on a project across seeds.
Each seed trains a separate model from scratch on a different
train/val/test split. Reads graphs from `data/processed/` and
saves model checkpoints back to `data/processed/`.

```bash
# Train on a single project (seeds 1 2 3 by default)
python scripts/train.py --project realm-java --epochs 1000

# Train on a single project with specific seeds
python scripts/train.py --project activemq --epochs 1000 --seeds 3 4 5

# Train on all projects
python scripts/train.py --all --epochs 1000

# Prevent laptop from sleeping during training (Mac)
caffeinate -i python scripts/train.py --project realm-java \
    --epochs 1000 2>&1 | tee results/realm_java_log.txt
```

Model architecture:
- **GraphSAGE encoder** — 2 layers, hidden dim 256, dropout 0.1
- **GraphSMOTE augmenter** — generates synthetic smelly nodes to
  handle 3-10% class imbalance
- **Node classifier** — linear layer + softmax with weighted
  cross-entropy loss
- **Edge generator** — predicts calling strength between methods
  for refactoring recommendation

Training hyperparameters:

| Parameter | Value |
|-----------|-------|
| Hidden dimension | 256 |
| GNN layers | 2 |
| Dropout | 0.1 |
| Learning rate | 1e-3 |
| Weight decay | 5e-4 |
| Lambda edge | 1e-6 |
| Epochs | 1000 |
| Seeds | 1, 2, 3 |

Checkpoints are saved automatically to
`data/processed/<project>/seed_<n>/checkpoint/`.

---

### Step 5 — Compute Final Results

Loads saved checkpoints and computes aggregated results across
all seeds without retraining. This script can be run any time
after training finishes.

```bash
# Compute results for a single project
python scripts/compute_results.py --project realm-java --seeds 1 2 3

# Compute results for activemq which used seeds 3 4 5
python scripts/compute_results.py --project activemq --seeds 3 4 5

# Compute results for all projects and save to JSON
python scripts/compute_results.py --all --seeds 1 2 3 \
    --output results/final_results.json
```

Results are saved to `results/gnn_results.json`.

in order:


## Key Design Decisions

**Why GNNs?** Feature envy is a relational smell — it is defined by
how a method relates to other methods and classes. GNNs operate
directly on graph structure, capturing interaction patterns that
simple metrics cannot.

**Why GraphSMOTE?** Smelly methods are rare (3-10% of all methods).
Without imbalance handling, the model collapses to predicting
all-negative. GraphSMOTE generates synthetic smelly nodes in the
graph embedding space to balance training.

**Why stratified splits?** With low positive rates, random splits
can produce training sets with almost no positive examples.
Stratified splitting preserves the positive rate across
train/val/test.

**Why 3 seeds?** The paper uses 5 seeds to measure result stability
across different data splits. We use 3 seeds due to computational
constraints of CPU-only training, which still provides a meaningful
measure of variance while reducing total training time from
approximately 8-10 hours to approximately 4-5 hours.

**Why separate preprocess and train scripts?** Preprocessing is
expensive and only needs to run once. Separating it from training
means graph construction artifacts are saved to disk and reused
across all training runs, avoiding redundant computation.

## Troubleshooting

**NumPy version conflict:**
```bash
pip install "numpy<2"
```

**torch-scatter/torch-sparse installation fails on Mac:**
```bash
# These are bundled in torch-geometric for Python 3.12+
# Just install torch-geometric directly
pip install torch-geometric
```

**Graph not found error:**
```bash
# Preprocessing has not been run yet
python scripts/preprocess.py --all
```

**Checkpoint not found error:**
```bash
# Training has not been run yet for this project/seed
python scripts/train.py --project <project> --seeds <seeds>
```

**Process interrupted during training:**
```bash
# Use caffeinate to prevent sleep on Mac
caffeinate -i python scripts/train.py --project <project> --epochs 1000

# Resume from a specific seed if earlier seeds completed
python scripts/train.py --project <project> --seeds 3 --epochs 1000
```
